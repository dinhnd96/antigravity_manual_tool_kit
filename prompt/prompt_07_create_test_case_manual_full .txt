# QUY TẮC “KHÔNG TRÙNG” + GỘP (DEDUP & MERGE ENGINE)
- **Không trùng** giữa TC-BR và TC-UI:
  • TC-BR tập trung vào **điều kiện/logic** (mandatory, auto-code, hiệu lực, auto-status, audit…), có thể chạm UI nhưng chỉ để kích hoạt BR. 
  • TC-UI **không** kiểm lại các BR; chỉ kiểm **luồng và hành vi UI bề mặt**: điều hướng, mở/đóng form, button state enabled/disabled, hiển thị trên lưới, tải xuống, pagination, filter syntax, encode file…
- **Gộp kiểm thử** khi có thể: nếu nhiều trường hợp là **biến thể tương tự** của **cùng một mục tiêu kiểm thử**, hãy:
  • Tạo **1 TC duy nhất** và mô tả **nhiều biến thể** trong phần Steps (Step 3a/3b/3c…) thay vì tạo nhiều TC rời. 
  • Chỉ tách TC khi: (i) khác BR, hoặc (ii) khác luồng UI rõ rệt (ví dụ “Thêm mới” vs “Sửa”), hoặc (iii) cần data setup/phân quyền rất khác.
- **Note only**: nếu một ý nghĩa **đã được cover** bởi TC khác, KHÔNG tạo TC mới; thay vào đó, thêm **1 dòng Note only**:
  “ĐÃ COVER ở TC_ID=<<ID>> (Module=<<...>>, URD_Ref=<<...>>, BR_Ref=<<...>>)”.

# PHÂN LOẠI & ƯU TIÊN
- Type: Happy / Boundary / Negative / Integration / Calculation / Security / Job / Report.
- Category: Smoke (10–20 TC trọng yếu) vs Regression (còn lại).
- Priority: P1 (Smoke/đường găng), P2, P3.

# HƯỚNG DẪN VIẾT TC (ÁP DỤNG CHO CẢ BR & UI)
- Precondition nêu rõ: vai trò (Maker/Checker), tham số hệ thống liên quan, data setup (nếu có), ngày T.
- Steps: 4–8 bước/TC, đánh số rõ, nếu có biến thể ghi 3a/3b/3c… (để **gộp**).
- Expected: gồm 4 lớp (tuỳ TC): 
  (i) Nghiệp vụ/logic, (ii) UI (toast/popup/lưới), (iii) trạng thái//audit/log, (iv) file/email (nếu có).
- URD_Ref: chỉ định đúng tiểu mục (ví dụ “I.1.1.1.2”).
- BR_Ref: 
  • Với **TC-BR**: ghi đúng mã BR (BR_01..BR_xx). 
  • Với **TC-UI**: ghi “UI-FUNC.<SỐ_CHỨC_NĂNG>” (vd: UI-FUNC.05 cho Lọc), **không** ghi BR để tránh chồng lặp. 
- Trace_ID: chuỗi ngắn phục vụ traceability (vd: “BR03-AUTOCODE”, “UI-FILTER-BASIC”).
- Không tạo, không gắn **version record mới** nếu TC chỉ là sửa hiệu lực: luôn qua **OT.01** (đúng Versioning/OT=C).

  • (A) Đếm TC theo BR (BR_01..BR_xx): Covered / GAP, mỗi GAP gợi ý 1 skeleton TC.

- Sheet “Dedup_Log”:
  • Liệt kê các cặp TC có khả năng trùng nội dung; nêu quyết định: GỘP vào TC_ID nào, và “Note only” ở đâu.
